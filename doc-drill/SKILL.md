# doc-drill

You have access to structured, hierarchical documentation generated by autoDoc for one or more codebases. This documentation is organized as a tree of progressively detailed layers. **Do not load everything at once** — drill down layer by layer, guided by relevance to the task at hand.

## Documentation structure

```
docs/{project}/
├── top.json              # Level 0: project overview + top-level modules
├── {Module}/
│   ├── {Module}.json     # Level 1: module graph with child nodes
│   ├── {Leaf}.md         # Leaf: detailed documentation page
│   └── {SubModule}/
│       ├── {SubModule}.json   # Level 2+: deeper sub-graph
│       └── ...
```

Each `.json` graph file contains:
- `description`: what this module does
- `codeScope`: which source files/directories it covers
- `nodes[]`: child nodes, each with:
  - `name`, `description`, `codeScope`
  - `edges[]`: relationships to sibling nodes (`calls`, `depends`, `data-flow`, `event`, `extends`, `composes`)
  - `child.type`: `"graph"` (has sub-graph, can drill deeper) or `"page"` (leaf, has `.md` doc)
  - `child.ref`: path to the child resource

Each `.md` leaf page contains detailed technical documentation for that component.

## The browse tool

Use `scripts/browse.mjs` (located in the doc-drill skill directory) to navigate the documentation:

```bash
# Step 1: Get project overview — see all top-level modules
node {skill-dir}/scripts/browse.mjs {skill-dir}/docs {project}

# Step 2: Drill into a module — see its children and relationships
node {skill-dir}/scripts/browse.mjs {skill-dir}/docs {project} {ModuleName}

# Step 3: Keep drilling into sub-modules
node {skill-dir}/scripts/browse.mjs {skill-dir}/docs {project} {Module}/{SubModule}

# Step 4: Read a leaf page when you've found the right component
node {skill-dir}/scripts/browse.mjs {skill-dir}/docs {project} {Module}/{LeafName} --read

# Search: Find nodes by keyword across all levels
node {skill-dir}/scripts/browse.mjs {skill-dir}/docs {project} --search {keyword}
```

Replace `{skill-dir}` with the actual path to the doc-drill directory.

## Workflow: progressive disclosure protocol

Follow this discipline every time you need codebase context:

### 1. Orient (top-level)
Run the browse tool with just the project name. Read the module list, their descriptions, and their `edges` to understand the high-level architecture. **Stop here if the task only needs architectural understanding.**

### 2. Locate (module-level)
Identify which 1-2 modules are most relevant based on descriptions and `codeScope`. Drill into those modules. Read child node descriptions and edges. **Stop here if you can now identify the right component.**

### 3. Focus (leaf-level)
Once you've identified the specific component, read its `.md` page with `--read`. This gives you detailed implementation docs including file-by-file breakdowns, data flows, and key APIs.

### 4. Search (when unsure)
If you can't tell which module to drill into, use `--search` with a keyword. This scans all levels and returns matching nodes with their module paths, letting you jump directly to the right depth.

## Key principles

- **Lazy loading**: Never read leaf `.md` files until you've confirmed they're relevant through the graph hierarchy. Each `.md` can be 5-15KB — loading all of them wastes context.
- **Edge-guided navigation**: The `edges` array on each node tells you which siblings interact. If you're tracing a data flow or call chain, follow the edges rather than reading every child.
- **codeScope as ground truth**: When you need to map documentation back to actual source files, use the `codeScope` field. It tells you exactly which files/directories a node documents.
- **Breadth before depth**: At each level, scan all node descriptions first to pick the right one, rather than guessing and drilling into the first match.

## Output conventions

When referencing documentation findings in your response:
- Cite the module path: `Core/SessionEngine/TurnLifecycle`
- Mention the relevant source files from `codeScope` when the user needs to edit code
- If a question spans multiple modules, trace the edges between them to explain the connection
